# Browser Keygen Onboarding — Implementation Plan

> **Goal:** Zero-friction first login. User opens their Continuum domain in a browser, generates a Nostr key in-browser, and is immediately logged in. No extension required. Optional migration to NIP-07 extension or NIP-46 bunker later.

## Architecture

### The Setup Wizard Flow

```
┌──────────────────────────────────────────────────────────────┐
│  ANSIBLE DEPLOY                                              │
│  1. Install system, agent, frontend, Caddy                   │
│  2. Agent starts in "first-run mode"                         │
│     - config: setup_mode: true                               │
│     - config: setup_token: <random 32-char hex>              │
│     - config: admin_npubs: [] (empty — not yet set)          │
│  3. Caddy proxies to agent (publicly reachable)              │
│  4. Ansible prints:                                          │
│     "Open https://your-domain in your browser."              │
│     "Your setup token: a1b2c3d4e5f6..."                      │
└──────────────────────────────────────────────────────────────┘
          │
          ▼
┌──────────────────────────────────────────────────────────────┐
│  USER OPENS BROWSER                                          │
│  Agent serves setup wizard (first-run mode)                  │
│                                                              │
│  Step 1: Enter setup token                                   │
│    POST /api/setup/verify-token { token }                    │
│    Agent verifies token matches config.setup_token            │
│                                                              │
│  Step 2: Browser generates key (Web Crypto)                  │
│    - 32 random bytes → nsec                                 │
│    - Derive pubkey → npub                                    │
│    - Show nsec + backup phrase to user                       │
│    - User confirms they saved it                             │
│                                                              │
│  Step 3: Register pubkey with agent                          │
│    POST /api/setup/register { pubkey, signed_event }          │
│    Agent verifies: token valid + signature valid              │
│    Agent writes pubkey to config: admin_npubs = [npub]        │
│    Agent sets: setup_mode = false                             │
│    Agent restarts (or hot-reloads config)                     │
│                                                              │
│  Step 4: Login with new key                                  │
│    Browser uses NIP-07 shim (signs from IndexedDB)            │
│    Normal challenge → verify → session token                  │
│    User is logged in. Done.                                  │
└──────────────────────────────────────────────────────────────┘
```

### Why This Works

- **Bot race eliminated:** Setup token required. Bots scanning the web don't have it.
- **Self-custody from t=0:** Key generated in browser, never touches VPS.
- **No extension needed:** NIP-07 shim signs from IndexedDB.
- **Optional upgrade:** User can export key to extension or connect NIP-46 bunker later.

### Key Management Priority

```
Is window.nostr present? (extension/Amber)
  ├─ YES → Use extension. Best path.
  └─ NO → Is there a key in IndexedDB?
           ├─ YES → Use NIP-07 shim (browser keygen key).
           └─ NO → Show setup wizard (first run) or
                    "connect signer" screen.
```

## Implementation Tasks

### Task 1: Agent — First-Run Mode + Setup Endpoints
**File:** `agent/core/config.mjs`, `agent/index.mjs`  
**Effort:** 4h

Config additions:
```yaml
# config.yaml — first-run state
setup_mode: true           # false after setup completes
setup_token: "a1b2c3..."   # random 32-char hex, generated by Ansible
admin_npubs: []            # empty until setup wizard fills it
```

New agent endpoints (only active when `setup_mode: true`):
```
GET  /api/setup/status     → { setup_mode: bool }
POST /api/setup/verify     → { token } → { ok: bool }
POST /api/setup/register   → { pubkey, signed_event } → { ok, session_token }
```

`/api/setup/register` flow:
1. Verify setup_token matches
2. Verify signed_event (kind 22242, correct pubkey, valid sig)
3. Write pubkey to admin_npubs in config
4. Set setup_mode = false
5. Issue session token (user is immediately logged in)
6. Return { ok: true, token, expires_at }

After setup_mode = false, setup endpoints return 404.

**Acceptance:**
- [ ] Agent starts correctly with empty admin_npubs in setup_mode
- [ ] Setup endpoints reject requests when setup_mode = false
- [ ] Token verification works
- [ ] Pubkey registration writes to config and exits setup mode
- [ ] Session token issued on registration

```javascript
  // Export encrypted backup
  async exportEncrypted(password) { ... }

  // Delete key from IndexedDB
  async delete() { ... }
}
```

NIP-07 shim in `src/auth.js`:
```javascript
// If no extension detected but KeyVault has a key,
// inject window.nostr shim that signs from KeyVault
if (!window.nostr && await KeyVault.hasKey()) {
  const vault = await KeyVault.load(sessionPassword);
  window.nostr = {
    getPublicKey: () => vault.getPubkey(),
    signEvent: (event) => vault.signEvent(event),
  };
}
```

## Security Model

| Threat | Mitigation |
|--------|------------|
| Bot races to claim instance | Setup token required (printed by Ansible, not on web) |
| XSS exfiltrates key | CSP headers, key only in memory when signing, IndexedDB encrypted |
| User loses browser data | Mandatory backup step (nsec + mnemonic shown during setup) |
| VPS compromised | Key never on VPS. Setup token rotated after use. |
| MITM during setup | HTTPS (Caddy auto-TLS). Setup token is one-use. |

## Dependencies

New npm packages:
- `@noble/secp256k1` — secp256k1 key generation + signing
- `@scure/base` — bech32 (nsec/npub encoding)
- `@noble/hashes` — SHA-256 for event IDs
- `@noble/ciphers` — AES-256-GCM for encrypted backup

All are audited, zero-dependency, browser-native libraries from the Paul Miller / nayuki ecosystem.
